Era 1: On-Premise (1950s-2000s)

Mainframes (1950s-1980s)

  • Giant centralized computers
  • Prohibitive costs (millions of dollars)
  • Access via dumb terminals
  • Examples: IBM System/360, VAX

Dedicated Physical Servers (1990s-2000s)

Characteristics

• Corporate data centers

• Rack-mounted servers

• One application per server

Challenges

• Massive upfront investment (CAPEX)

• Long procurement cycles (months)

• 10-15% average utilization

• Constant physical maintenance

Example: Buying servers for Black Friday peaks that sat idle 364 days a year

Era 2: Virtualization (2000s)

The Virtual Machine Revolution

1999

VMware ESX launched

2003

Xen hypervisor released

2008

Microsoft Hyper-V introduced

The Breakthrough

✓ Advantages

• Multiple VMs per physical server

• Application isolation

• Better resource usage (40-60%)

• Faster provisioning (hours vs weeks)

• Snapshots and cloning

✗ Limitations

• Still requires OS management

• Hypervisor overhead

• Heavy VMs (GBs in size)

• Boot time in minutes

  • Self-service: APIs and web consoles
  • Elasticity: Scale up/down on demand

Era 4: Containers (2013-2015)

"Docker launched in 2013: The Container Revolution"

What are Containers?

  • Application + dependencies packaged together
  • Share host OS kernel
  • Much lighter than VMs (MBs vs GBs)
  • Start in seconds

VM vs Container Comparison

Virtual Machine

  • Includes full Guest OS
  • GBs in size
  • Minutes to boot
  • Hardware virtualization

Container

  • Shares host OS
  • MBs in size
  • Seconds to start
  • OS-level virtualization

Kubernetes: The Orchestrator

Features

• Declarative management

• Auto-healing and auto-scaling

• Service discovery

• Rolling updates

Challenges

• Cluster management complexity

• Kubernetes learning curve

• Capacity planning still needed

Era 6: Serverless / FaaS (2014-Present)

"AWS Lambda - November 2014: The Birth of Serverless
Don't think about servers. Think about functions executing in response to events."

Revolutionary Characteristics

1. Ephemeral Execution

• Functions live only during execution

• No 24/7 infrastructure

• Infinite transparent auto-scaling

2. Event-Driven

• Responds to events from multiple sources

• Native cloud ecosystem integration

• Event-oriented architecture

3. Micro-billing

• Charged per 100ms of execution

• First 1M invocations free (AWS)

• Zero cost when not executing

4. Zero Administration

• No servers to manage

• No patching or maintenance

• No capacity planning

What is Serverless?

"Serverless doesn't mean no servers, but rather that developers don't manage servers"

Key Principles

  • Developers don't manage servers: Infrastructure is completely abstracted
  • Provider handles everything: Provisioning, scaling, and maintenance
  • Pay-per-use: Billing based on actual consumption

Main Components

Function as a Service (FaaS)

• AWS Lambda, Azure Functions, Google Cloud Functions

• Code units executed on demand

• Stateless and ephemeral

• Respond to specific events

Backend as a Service (BaaS)

• Managed services complementing functions

• Databases (DynamoDB, Firestore)

• Authentication (Cognito, Auth0)

• Storage (S3, Cloud Storage)

• Managed APIs (API Gateway)

Economic Benefits

Reduced Operational Costs

When NOT to Use Serverless

✗ Unsuitable Scenarios

Monolithic Legacy Apps

• Difficult to decompose into functions

• High coupling

• Better to migrate gradually

24/7 Constant Workloads

• Continuous predictable high usage

• May be more expensive than reserved instances

• Calculate TCO before deciding

Specific Hardware Requirements

• Specialized GPUs

• FPGAs

• Custom hardware

Heavy Dependencies

• Applications with giant libraries (>250MB)

• Very large containers

• Unacceptable cold start time

⚠ Consider Carefully

  • Critical Cold Starts: Apps with very tight latency SLA (<100ms) - Consider Provisioned Concurrency
  • Long Processing: Tasks over 15 minutes (Lambda limit) - Alternative: ECS Fargate, Batch
  • Complex Shared State: Strongly stateful applications - Solution: Use cache services (ElastiCache, DynamoDB)

Challenges and Considerations

Technical Challenges

Debugging & Observability

Challenge: Distributed logs, complex tracing

Solution: CloudWatch Logs, X-Ray, structured logging, distributed tracing

Vendor Lock-in

Challenge: Dependence on provider-specific services

Mitigation: Multi-cloud frameworks, service abstraction layers

Execution Limits

Constraints: Timeout (15min), Payload size (6MB), Memory (10GB)

Solution: Design within limits or use alternatives

Local Testing

Challenge: Different environment from production

Solution: SAM CLI, LocalStack, Serverless Framework

Organizational Challenges

  • Mindset Shift: Distributed architecture, event-driven thinking, need for training
  • Cost Monitoring: Difficult to predict costs initially, need for tagging and alerts, continuous optimization

Best Practices: Function Design

Design Principles

Single Responsibility

• One function, one specific task

• Small and focused functions

• Facilitates maintenance and testing

Idempotency

• Functions produce same result with same input

• Crucial for retry logic

• Avoids duplicate processing

Statelessness

• Don't maintain state between invocations

• Use external services for persistence

• Cache connections in global scope

Performance Optimization

  • Cold Start Optimization: Minimize dependencies, lazy loading of resources, Provisioned Concurrency for critical cases
  • Memory Sizing: More memory = more CPU, test different configurations, balance cost vs performance
  • Connection Reuse: Database connections outside handler, reuse SDK clients, connection pooling

Best Practices: Security

Least Privilege

• IAM roles with minimum permissions

• One role per function when needed

• Review permissions regularly

• Use resource-based policies

Secrets Management

• Never hardcode credentials

• AWS Secrets Manager or Parameter Store

• Automatic secret rotation

• Encrypt at rest and in transit

Input Validation

• Always validate input data

• Payload sanitization

• Protection against injection attacks

• Use schema validation

Network Security

• VPC integration when needed

• Security groups and NACLs

• Private subnets for sensitive resources

• VPC endpoints for AWS services
